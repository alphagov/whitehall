******************************************************************************************
Summary
******************************************************************************************
Legend: SO = slug override; SFT = slug from title; CIP = Corporate Information Page; LP = Landing Page; WNS = World News Story

The latest migration(s) we ran populated the SFT column with a title-based slug, but also set the slug, for all pre-publication editions. This was slightly overzealous, as at this point in time the code does not (correctly) set the slug column for new editions, just the SFT.

Because this change did more than what we bargained for, but at a point in time where the code wasn't ready to handle it, we took a step back to validate some special cases, just to ensure that the migration did not cause any damage, and to understand the landscape of the data a bit better before we move forward with any further changes.

There are so many moving factors here: 3 columns about slugs, two readers (slug and slug_from_title) changing what is returned in the code, the checkbox flipping values, under certain conditions, and all of that compounded by the editionable flow. On top of this there are special slug cases:
- CIPs - have their own slug definition, and exit early in the set_slug method.
- non-en primary locale editions - have different slug behaviours; these behaviours are even different between regular editions and config-driven editions. WNSs in particular had unexpected slugs set, and the checkbox behaviour was tricky to understand for these.
- suffixed slugs seemed potentially problematic
- Landing pages - TBC

Overall, understanding what also makes sense from a data integrity perspective was also necessary. Leaving the slug columns with very inconsistent values, across different states and types, is not ideal, especially as we migrate types to config-driven. We hoped that understanding the data would help make better calls about the logic we want to implement, and the migrations we want to run, going forward. Overall adding a third column (i.e. slug_from_title) will allow us to manage the behaviours better, and manipulate the checkbox rendering.

For context, the history of these changes is:
- add slug column to editions
- backfill slug column from document ID
- add slug_override column to editions
- set slug_override where edition slug was different from the document slug
- remove document slug column
- add a slug_from_title column to editions
- run migration to backfill slug_from_title from title (where we also set slug to the value of either override or SFT)

3.1. Special case - scheduled
- Editions get scheduled with their ID as a job arg. If an edition was scheduled prior to migration, to run after migration, with the setting to use a title based slug, and a different slug was generated by our migration (it runs the set_slug callback without a title change), then would the scheduled job use the new slug? Yes it would, since the job only has the edition ID. Nonetheless, I am fairly confident that the slug crash is not possible because the slug candidate generator checks by document ID. Running the set_slug is fairly idempotent.

3.2. Translations
- Proper translations of another primary locale don't get custom slugs. This is the same regardless of the type (edition, standard edition).

    - no draft edition slugs (that have suffixes) correspond to live editions with slug overrides (though some cases look like they should, because the titles do not match the slug on the live)
    - where slug overrides are set on the drafts, the slug = SO
    - where slug override is not set - it is likely that the user changed the title and opted to use the new slug (even if it's suffixed).
